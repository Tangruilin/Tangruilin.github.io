leveldb中的版本控制

发表于 2025-02-22
本文字数： 886 阅读时长 ≈ 3 分钟

背景

版本控制和元信息管理, 在任何系统中都有着举足轻重的地。leveldb 中为了更好地做并发支持, 单独抽象出 Version 模块; 本文会整体梳理 leveldb 中 Version 模块的作用;

作用

leveldb 作为一个有着持久化能力的 LSM 数据库, 需要知道每一个 level 中都包含了哪些 LSM 文件; 重启后如何快速恢复到之前的状态; 这些能力都依赖于元数据管理模块, 在 leveldb 中, Version 模块则起到的下面几个作用:

  1. 维护 SST 文件的索引层次信息, 记录了在 leveldb 中每一个 SST 文件所在的索引位置, 为读, 写，Compaction 等操作提供元数据支持;
  2. 记录 Compaction 相关信息, 保证 Compaction 能在需要的时候被触发;
  3. 保证数据库在宕机重启的情况下, 数据的信息可以不丢失;
  4. 维护仍在使用的历史 Version 信息, 保证在进行 Compaction 的时候, 仍在被读的 sst 文件不被删除, 简化并发控制;

实现

leveldb 中, 为了实现元数据管理, 主要有三个核心的类: Version, VersionSet, VersionEdit, 他们的作用分别是:

  1. Version。Version 用来表示单个 leveldb 的某一个数据版本的元数据信息;
  2. VersionSet。VersionSet 表示整个 leveldb 数据库的所有元数据, 包含了多个 Version;
  3. VersionEdit。VersionEdit 更应该理解为 VersionDelta, 通常会用于 Version 发生变更的场景;

VersionSet 和 Version 的设计

VersionSet 和 Version 的整体布局大致如下图所示:


VersionSet 表示整个数据库的所有存在的版本, Version 则静态表示一个版本, Version 之间会通过双向链表进行关联, 同时每一个 version 由 refs 来维持其引用计数, 当计数为 0 时, 该 version 自动从 VersionSet 中剔除, 其指向的数据文件也会在后台被 GC。
值得一提的是, VersionSet 中的 prev_log_num 在当前版本中已经废弃, 只是为了兼容老版本的持久化文件(manifest 文件的编解码)才被保留

VersionEdit 的设计

VersionEdit 更应该叫做 VersionDelta, 理解这个结构体只需要记住 Version + VersionEdit = Version + 1 则可, 在使用上, VersionEdit 不会做实际的落盘动作(换句话说, 保存在 VersionEdit 中的数据都是尚未持久化的), 在 VersionSet::Apply 和 VersionSet::LogAndApply中, 变化的部分会被更新, 同时只有 VersionSet::LogAndApply中, 才会对 Version 内容做持久化。

总结和思考

对于整个 Version 模块的设计和演进, 整体上有三点可以思考的地方:

  1. 软件系统的对象化思维, 在关系数据库中有通用的 Catalog 设计来做元数据的管理, leveldb这类 kv 数据库则自行设计了相对简单的 Version 模块, 这里体现了面向对象的思路;
  2. 对于对象的变化处理。对于一个状态机(这里的 Version 就是一个状态机), 状态发生改变时, 其状态会由 改变前 –> 改变中 –> 改变后, 将模块设计为 base + delta 的方式, 可以有效避免频繁的状态改变带来版本膨胀;
  3. 由于一个存储系统在完成后通常会需要服务多年, 其向前兼容性与向后兼容性一定要做充分的考虑。
